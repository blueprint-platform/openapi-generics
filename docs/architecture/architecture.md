## Architecture Overview

openapi-generics is built around a simple principle:

> Java contract is the source of truth. OpenAPI is a projection. Generated clients are reconstructed from that projection.

The system consists of two independent execution surfaces:

```text
Spring Boot Service
        │
        ▼
openapi-generics-server-starter
        │
        ▼
OpenAPI + Vendor Extensions
        │
        ▼
openapi-generics-java-codegen-parent
        │
        ▼
GenericAwareJavaCodegen
        │
        ▼
Generated Java Client
```

The OpenAPI document is the only artifact shared between server-side and client-side processing.

No runtime state is shared.

No direct coupling exists between server and client generation.

## Projection Protocol

Generic semantics are carried through a small set of vendor extensions.

### Wrapper Metadata

```yaml
x-api-wrapper: true
x-api-wrapper-datatype: CustomerDto
```

Identifies a generated wrapper schema and its payload type.

### Container Metadata

```yaml
x-data-container: Page
x-data-item: CustomerDto
```

Represents container-aware payloads such as:

```java
ServiceResponse<Page<CustomerDto>>
```

or

```java
ServiceResponse<List<CustomerDto>>
```

### Generation Control

```yaml
x-ignore-model: true
```

Marks schemas that must not be regenerated by client generators.

Examples:

```text
ServiceResponse
Meta
Sort
Page
```

The server publishes these extensions.

The client generator consumes them.

Together they form the projection protocol used by openapi-generics.

## Server Pipeline

The server starter projects Java response contracts into OpenAPI.

Pipeline:

```text
Controller Return Types
        ↓
Response Discovery
        ↓
Contract Introspection
        ↓
Wrapper Projection
        ↓
Container Enrichment
        ↓
Generation Control
        ↓
Contract Validation
```

Core responsibilities:

### Response Discovery

Collect supported controller response types from the application.

### Contract Introspection

Extract semantic response descriptors from:

```java
ServiceResponse<CustomerDto>
ServiceResponse<Page<CustomerDto>>
ServiceResponse<List<CustomerDto>>
```

or:

```java
ApiResponse<CustomerDto>
```

for BYOE scenarios.

### Wrapper Projection

Create deterministic wrapper schemas:

```text
ServiceResponseCustomerDto
ServiceResponsePageCustomerDto
ServiceResponseListCustomerDto
```

### Container Enrichment

Attach container metadata required for client reconstruction.

### Generation Control

Prevent regeneration of platform-owned schemas.

### Contract Validation

Verify that projected schemas remain internally consistent.

Invalid projections fail startup immediately.

## Build Pipeline

The build-side orchestration is provided by:

```text
openapi-generics-java-codegen-parent
```

Pipeline:

```text
OpenAPI
      ↓
Template Preparation
      ↓
Template Overlay
      ↓
OpenAPI Generator
      ↓
GenericAwareJavaCodegen
      ↓
Generated Sources
```

The parent module owns:

- template extraction
- template patching
- template overlay
- OpenAPI Generator integration
- generated source registration

The codegen module does not own build orchestration.

## Code Generation

`GenericAwareJavaCodegen` extends the standard Java generator with contract awareness.

Responsibilities:

### Ownership Resolution

Determine whether a model should:

```text
Generate
or
Reuse Existing Type
```

### Model Filtering

Exclude:

- ignored schemas
- external contract models
- platform-owned infrastructure models

### Metadata Enrichment

Inject:

```text
Envelope metadata
Container metadata
External import metadata
```

for wrapper generation.

### Wrapper Reconstruction

Generate thin wrapper types:

```java
public class ServiceResponseCustomerDto
    extends ServiceResponse<CustomerDto> {
}
```

```java
public class ServiceResponsePageCustomerDto
    extends ServiceResponse<Page<CustomerDto>> {
}
```

```java
public class ApiResponseCustomerDto
    extends ApiResponse<CustomerDto> {
}
```

Generated wrappers preserve type structure.

They intentionally contain no behavior.

## Determinism & Validation

The platform is designed around deterministic output.

Guarantees:

```text
Same Contract
      ↓
Same OpenAPI
      ↓
Same Generated Client
```

Validation occurs at multiple stages:

### Server Side

- envelope validation
- contract validation
- projection validation

### Build Side

- template validation
- OpenAPI validation
- generation validation

The system prefers fail-fast behavior.

Principle:

> Incorrect generation is worse than no generation.

## Supported Boundaries

Supported response shapes:

```java
ServiceResponse<T>
ServiceResponse<Page<T>>
ServiceResponse<List<T>>
```

BYOE support:

```java
YourEnvelope<T>
```

Custom envelopes must:

- be concrete classes
- declare exactly one type parameter
- expose exactly one direct payload field

Not supported:

```java
YourEnvelope<Page<T>>
YourEnvelope<List<T>>
Map<K,V>
Nested Generic Graphs
```

The scope is intentionally constrained to keep projection deterministic.

